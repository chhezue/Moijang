## 1. 목표 플로우 (현행)

상세 설명: [01 §3](./01-PG-결제-아키텍처-와-Reservation-의사결정.md#3-목표-플로우), [02 §2](./02-Payment-Participant-오케스트레이션.md#2-호출-흐름)

```text
checkout → INITIATED + 선검증
confirm → PAID + join (실패 시 cancel + FAILED)
refund → REFUNDED + withdraw
leaderCancel → CANCELLED + refundForGroupBuyingCancellation (부분 실패 허용)
```

## 2. P0 (금전/정합성)

### 2.1 결제 소유자 검증

| 항목                               | 상태 | 근거 |
| ---------------------------------- | ---- | ---- |
| `confirm` `String(payment.userId)` | ✅   |      |
| `refund` 동일                      | ✅   |      |

### 2.2 confirm 멱등

| 항목                  | 상태 | 근거                                                        |
| --------------------- | ---- | ----------------------------------------------------------- |
| `PAID` 재호출         | ⚠️   | `'이미 결제가 완료되었습니다.'` 문자열 (participant 미조회) |
| participant 유실 정책 | ⬜   | 404 vs 재생성 vs 409                                        |

### 2.3 refund 멱등

| 항목                            | 상태 | 근거                            |
| ------------------------------- | ---- | ------------------------------- |
| `REFUNDED` early return         | ✅   | `'이미 환불이 완료되었습니다.'` |
| Swagger `Participant \| string` | ⚠️   | API 문서 보강 필요              |

## 5. 남은 핵심 리스크 (요약)

포트폴리오용 — 상세는 위 표 참고.

- confirm 멱등 시 participant 유실 정책 미정
- 일괄 환불 `partialSuccess` 후속 처리 없음
- 환불 후 모집 상태 역전 정책 없음
- 공구 취소 vs in-flight confirm — `RECRUITING` 재검증으로 완화, 극단 동시성은 MVP 밖
