# 04. 초기 PG 설계 아카이브 (Reservation 중심)

> **성격:** 과거 설계 기록 — **현행 운영 기준 아님**  
> **현행 단일 기준:** [01-PG-결제-아키텍처-와-Reservation-의사결정.md](./01-PG-결제-아키텍처-와-Reservation-의사결정.md)

---

## 1. 아카이브 유지 목적 (포트폴리오)

- 초기에 **왜 결제 전 선점(Reservation)을 검토했는지** 배경 추적
- Reservation **있음/없음** 설계 비교
- “복잡한 설계를 검토했다가 MVP에 맞게 단순화했다”는 의사결정 맥락 보존

---

## 2. 초기 설계 (아카이브)

| 항목         | 초기(Reservation) | 현행 (2026-06-01)             |
| ------------ | ----------------- | ----------------------------- |
| 결제 전 선점 | Reservation + TTL | **없음**                      |
| 정원 확정    | 선점 + confirm    | **confirm만**                 |
| 선점 만료    | TTL / Cron        | **불필요**                    |
| 동시성       | checkout 선점     | confirm 경쟁 + 환불           |
| 리더 무산    | (미정)            | cancel-orchestrator 일괄 환불 |

---

## 3. 현행과의 관계

**2026-06-01 이후** 운영 설계:

- Reservation 없이 `Payment` + confirm 시점 `Participant` 생성
- 리더 무산 시 `cancel-orchestrator` + `refundForGroupBuyingCancellation`
- `remainingCount` / Cron / TTL 예약 만료 — **미도입** (필요 시 [01 §2.7](./01-PG-결제-아키텍처-와-Reservation-의사결정.md#27-실무-확장-순서-필요할-때만))

상태 머신·엔드포인트·환경변수는 **01번 문서**를 따른다.

---

## 4. Reservation 폐기 근거 (요약)

자세한 트레이드오프: [01 §2](./01-PG-결제-아키텍처-와-Reservation-의사결정.md#2-reservation-제거-의사결정)

- 캠퍼스 공구 규모에서 동시 경쟁이 극단적이지 않음
- confirm + 자동 환불로 정합성 확보 가능
- 코드·운영 단순함 우선 (MVP)
